# Morning Briefing — Design Specification v1.4

**Owner:** Pat Harding, Surry Hills, Sydney
**Last updated:** June 2026
**Status:** Design complete. Build not started.

---

## Overview

An automated personal morning briefing system. Runs on a schedule each morning before Pat wakes up, pulls data from multiple personal data sources, synthesises a structured daily note, and delivers it as a formatted HTML email to Pat's Gmail via AWS SES.

Draft correspondence is generated by Claude and included in the email with one-click approve/reject links. No manual daily input is required beyond approving drafts.

**Estimated monthly cost: ~$1.50 AUD**

---

## Email Structure

Seven sections delivered as a single HTML email:

```
[Contextual opening line]

1. WEATHER
2. APPOINTMENTS & COMMITMENTS
3. TO-DO
4. FITNESS
5. DAILY 5
6. TODAY'S PHILOSOPHER
```

---

## Section Specifications

### Opening Line

A single contextual sentence from Claude Sonnet, generated as part of the synthesis step. Grounded in the actual day — calendar load, Garmin recovery, notable flags. Not generic.

> *"Good morning, Pat — tempo session today, two drafts to approve, and HRV is looking strong. Good day to push."*

---

### 1. Weather

- **Source:** OpenWeatherMap API (free tier)
- **Location:** Surry Hills, Sydney (−33.8885, 151.2099)
- **No LLM** — deterministic code and rule-based advisory flags
- Shows today, tonight, tomorrow
- Advisory line triggered by conditions during training window (06:00–09:00)

**Advisory thresholds:**

| Condition | Threshold |
|-----------|-----------|
| Rain | ≥ 2 mm |
| Heat | ≥ 28°C |
| Wind | ≥ 30 km/h |

---

### 2. Smart Date Awareness

Pure logic — zero LLM cost. Contextual flags injected into relevant sections.

**Race countdowns** (injected into Fitness section):

| Lead time | Flag |
|-----------|------|
| 8 weeks | Entering peak training block |
| 4 weeks | Final build week approaching |
| 2 weeks | Taper starts [date] |
| 1 week | Race week |
| Day of | Race day |

**School/parenting flags** (injected into Appointments section):
- School holidays starting within 3 days
- Last school drop-off of the week

**Birthday advance notice** (injected into Appointments section):
- Any birthday in the next 7 days that isn't today

**Fitness plan expiry** (injected into Fitness section):
- Warning when current training block ends within 7 days

---

### 3. Appointments & Commitments

- **Source:** Apple/Google Calendar API — appointments and birthdays
- **Source:** Parenting schedule JSON — today's responsibilities from fixed weekly pattern
- **No LLM** — deterministic code
- Exceptions in parenting JSON override the base weekly pattern
- School holiday date ranges stored in parenting JSON

---

### 4. To-Do

Three sub-sections:

**Due today** — from Apple Reminders API. Items due today or overdue. No LLM.

**Suggested actions** — Claude Sonnet infers actions from Gmail thread summaries. Examples: follow up on unanswered emails, schedule meetings referenced in correspondence, action items from school or professional emails.

**Correspondence to issue** — Claude Sonnet generates draft emails/replies. Each draft appears with [Approve] and [Reject] links. Approving triggers send via Gmail API.

---

### 5. Fitness

All data from Garmin Connect API and fitness plan YAML. No LLM except one Haiku call for a one-line recovery/feedback commentary.

**Displays:**
- Recovery status: HRV, sleep duration, Body Battery
- Today's planned session (from YAML, looked up by date)
- Last session summary (from Garmin — discipline, duration, distance, avg HR)
- Weekly session progress vs plan
- Weight goal: current vs 90 kg target (from Garmin)
- Sleep goal: 30-day average vs 7 hr target (from Garmin)
- Smart date race countdown flag if applicable
- Plan expiry warning if applicable

**Athlete context:**
- Disciplines: run, cycle, swim, brick, occasional strength
- All tracked via Garmin Connect (Garmin watch)
- 3-month periodised blocks (base → build → peak/race)
- 4–5 annual A/B events: marathons, triathlons, gran fondos
- Plans are static per month — Pat commits a new YAML file each month
- Claude provides feedback and suggested adjustments in the email but does not auto-modify the plan
- Known events: Sydney Marathon (Sep 2026, priority A), Huskisson Triathlon (Nov 2026, priority B)

---

### 6. Daily 5

- 5 web searches run as Batch API job at 11:00 pm the night before
- One search per topic bucket: World, Australia, Technology, Science, Business
- Topic buckets configurable in `briefing-config.json`
- Trusted sources only: Reuters, AP, BBC, The Guardian, ABC Australia, SMH, FT
- Last 24 hours only — older stories discarded
- Output: one sentence per bullet, maximum 25 words
- Results stored in S3, pulled into 5:00 am morning assembly
- Not payments-industry focused — Pat has a separate newsletter pipeline for that

---

### 7. Today's Philosopher

- **Source:** Claude Haiku
- A single quote selected based on a one-line context summary (sleep quality, commitments, notable flags)
- Closing section of the email
- Never forces a strained connection — defaults to consistent effort/showing up if context is unremarkable

**Traditions and authors:**

| Tradition | Authors |
|-----------|---------|
| Stoic | Marcus Aurelius, Epictetus, Seneca, Cato |
| Existentialist | Camus, Sartre, de Beauvoir, Heidegger, Kierkegaard |
| Moral philosophy | Aristotle, Kant, Mill, Wittgenstein, Bertrand Russell |

**Contextual mapping:**

| Context | Tradition/Author |
|---------|-----------------|
| Poor sleep, high load | Stoic — resilience, control |
| Many social commitments | de Beauvoir or Aristotle — relationships, obligation |
| Decisions pending | Sartre — responsibility, choice |
| Mundane or repetitive day | Camus — meaning in the everyday |
| Physical focus, strong training | Epictetus — discipline |
| Race approaching | Marcus Aurelius — focus, preparation |
| Quiet reflective day | Kierkegaard or Russell |

## Config Files

Three files maintained by Pat in a private GitHub repository. Read at runtime by the ingest Lambda.

```
personal-config/
  parenting-schedule.json    # Fixed weekly pattern + exceptions + school holidays
  fitness-plan.yaml          # Current 3-month training block
  briefing-config.json       # Location, weather thresholds, news topics, smart date rules
```

### parenting-schedule.json

Per-period schedule (morning / afternoon / evening / night) with named parties (pat, pam, sophie). Sophie is the Monday babysitter — arrives 16:00, handles pickup for both children, dog walk, dog feeding, and Philippa's dinner. Children: Penny (5, school) and Philippa (1, daycare).

See `personal-config/parenting-schedule.json` for the current schema and values.

### fitness-plan.yaml

Pat authors a new file each month. Claude Code assists with generation — Pat describes goals and the block, Claude produces the YAML, Pat reviews and commits.

Disciplines: `run`, `cycling`, `swim`, `brick`, `strength`

Key athlete fields: `lthr` (run), `threshold_pace`, `ftp` (cycling)

The system looks up today's session by matching the current date against `current_block.start` and calculating week/day position. Rest days are explicit or implied by absence of a session entry.

### briefing-config.json

See `personal-config/briefing-config.json` for the current schema and values.

## AWS Architecture

### Two-stage pipeline

**Stage 1 — 11:00 pm (night before)**
```
EventBridge → Batch Lambda
  Daily 5 news (Claude Sonnet, Batch API) → S3
```

**Stage 2 — 5:00 am (morning)**
```
EventBridge → Ingest Lambda (no LLM)
  Calendar API     → appointments, birthdays
  Parenting JSON   → today's responsibilities
  Reminders API    → open to-dos
  Garmin API       → HRV, sleep, weight, last activity
  Fitness YAML     → today's session
  Weather API      → forecast + advisory flags
  Smart date logic → race countdowns, school flags, birthday warnings
  → Pre-processed structured JSON → S3

→ Gmail Lambda (Claude Haiku)
  Classify threads — action needed or not
  Summarise relevant threads
  → S3

→ Synthesis Lambda (Claude Sonnet, prompt caching)
  Infer suggested actions from Gmail summaries
  Generate draft correspondence
  Select philosopher quote (Claude Haiku)
  Pull Daily 5 batch result from S3
  Generate fitness feedback commentary (Claude Haiku)
  Assemble final briefing JSON
  → S3

→ Email Builder Lambda (no LLM)
  Render HTML email from briefing JSON
  → SES → Gmail
```

### AWS Services

| Service | Purpose |
|---------|---------|
| EventBridge | Two cron triggers — 11:00 pm and 5:00 am AEST |
| Lambda | All compute — ingest, Gmail, synthesis, email builder, approval API |
| S3 | Raw ingest data, batch results, final briefing JSON |
| DynamoDB | Draft correspondence status tracking |
| SES | HTML email delivery to Pat's Gmail |
| API Gateway | Approve/reject endpoints for correspondence drafts |
| Secrets Manager | Garmin, Google, Anthropic, GitHub, OpenWeatherMap API keys |

| Suggested actions inference | Claude Sonnet |
| Briefing assembly + opening line | Claude Sonnet |
| Daily 5 news | Claude Sonnet + Batch API |

### Cost optimisations

- **Prompt caching** — all system prompts cached (~90% discount on cached input tokens)
- **Batch API** — Daily 5 runs at 11:00 pm, 50% discount
- **Pre-processing** — all data structured in code before any LLM call
- **Gmail filtering** — newsletters, marketing, automated mail excluded before any LLM call

**Estimated daily cost: ~$0.051 → ~$1.50 AUD/month**

## Messaging Architecture

The pipeline uses a normalised message schema so adding new sources (iMessage, WhatsApp, Messenger) requires only a new adapter — nothing downstream changes.

### NormalisedMessage

```typescript
interface NormalisedMessage {
  id: string
  source: 'gmail' | 'imessage' | 'whatsapp' | 'messenger'
  source_id: string
  thread_id: string
  direction: 'inbound' | 'outbound'
  sender: { name: string; identifier: string }
  recipients: Array<{ name: string; identifier: string }>
  subject?: string
  body: string
  timestamp: string
  read: boolean
  attachments?: Array<{ type: string; name: string }>
  metadata: Record<string, unknown>
}
```

### MessageAdapter

```typescript
interface MessageAdapter {
  source: MessageSource
  isEnabled(): boolean
  fetch(since: Date): Promise<NormalisedMessage[]>
}
```

Gmail adapter is active in v1. iMessage, WhatsApp, and Messenger adapter stubs are scaffolded and ready.

## Correspondence Approval Flow

1. Sonnet generates drafts during synthesis
2. Stored in DynamoDB with `status: 'draft'`
3. Appear in email with [Approve] / [Reject] links
4. Links hit API Gateway → Lambda
5. Approve → send via Gmail API, update status to `'sent'`
6. Reject → update status to `'rejected'`, discard

### DynamoDB draft schema

```typescript
{
  id: string
  created_at: string
  status: 'draft' | 'approved' | 'rejected' | 'sent'
  to: string
  subject: string
  body: string
  context: string
  approved_at?: string
  sent_at?: string
}
```

## Decisions Made

Do not re-open without good reason.

| Decision | Rationale |
|----------|-----------|
| Email delivery (not app, Slack, Notion) | Lowest friction, searchable, approval links work natively |
| Apple Reminders for to-dos (not Notes) | Notes has no API; Reminders has EventKit integration |
| Financials excluded from v1 | Requires Plaid or manual input — too much friction for v1 |
| Full OKRs excluded from v1 | Reduces manual maintenance burden; weight + sleep retained as auto-trackable goals |
| Weight and sleep goals retained | Both auto-tracked via Garmin — zero manual input |
| OKR section removed entirely | Goals surface inside the fitness section, not as a separate section |
| Static fitness plan per month | Pat prefers to author and control the plan; Claude provides feedback but doesn't auto-modify |
| Daily 5 not payments-focused | Pat has a separate newsletter pipeline for industry news |
| Exactly 5 web searches for Daily 5 | Cost control — one search per topic bucket, no follow-up fetches |
| One sentence max per news bullet (25 words) | Keeps the brief tight and fast to read |
| Philosopher quote: Stoic + Existentialist + Moral philosophy | Broader relevance than Stoic alone; contextually matched to the day |
| WhatsApp and Messenger not in v1 | No clean personal API access |
| Normalised message schema from day one | Avoids refactor when iMessage/WhatsApp added in v2/v3 |
| Audio briefing deferred to v2 | Good idea; backlogged to keep v1 scope tight |
| Friday edition deferred to v2 | Valuable but adds complexity — build once core is stable |

## V2 Roadmap

All discussed and deferred. None require re-evaluation from scratch.

- **iMessage via Mac daemon** — adapter stub in repo, daemon package scaffolded, enable via config flag
- **WhatsApp + Messenger via Mac daemon** — same pattern as iMessage, ToS review required
- **Audio briefing** — AWS Polly (simplest) or ElevenLabs (better quality); private podcast feed recommended long-term
- **Full OKR tracking** — JSON schema fully designed; ready to introduce
- **Financials** — bills, budget, savings via Plaid or manual CSV
- **Adaptive fitness planning** — Claude adjusts next week based on Garmin training load
- **Monthly fitness summary email** — separate email with volume charts and plan suggestions
- **Reply detection** — surface replies to approved drafts in next morning's email
- **Friday edition** — weekly lookback format on Fridays

## Recommended Build Sequence

1. Ingest layer — Calendar, Garmin, Reminders, Weather → structured JSON to S3
2. Email builder — render HTML from structured JSON, send via SES
3. Fitness section — YAML plan + Garmin data, validate the session display
4. Daily 5 — Batch job, web search, validate news bullets
5. Gmail classification and suggested actions — Haiku classification
6. Draft correspondence generation and approval flow
7. Philosopher quote — Haiku contextual selection
8. Smart date awareness — rule-based flags
9. Opening line — Sonnet synthesis
